# DevLauncher Cloud Sync Design

## Goal

Add a first cloud sync path for personal device migration. A user can upload the current local DevLauncher configuration to a private server-backed PostgreSQL database, then enter a sync key on a new computer and restore the latest snapshot with one explicit action.

The first version is a manual backup and restore workflow, not realtime bidirectional sync.

## Scope

Sync these local data sets:

- `keyboard.yaml`, including pages, key bindings, theme, pet Codex setting, and pet custom menu actions.
- `quickmemory_data.json`, including custom QuickMemory categories, custom items, order, and copy counts.

Do not sync these data sets in the first version:

- Real SSH, web, remote desktop, or other passwords stored in the OS credential store.
- Private keys, token files, script target files, local application bundles, or folder contents referenced by bindings.
- Browser `localStorage` history for screenshot, JSON helper, API Lab, or other plugin-specific transient state.

`hasPassword` flags may remain in `keyboard.yaml`, but restore must treat them as metadata only. On a new computer, the UI should clearly show that passwords need to be re-entered.

## Product Behavior

Settings gets a new "Cloud Sync" section.

The section contains:

- Connection state: disconnected, connected, upload failed, restore failed.
- Sync key input.
- Save sync key locally.
- Upload current configuration.
- Restore latest cloud snapshot.
- Latest remote snapshot time, device name, and app version when available.

Upload creates a new immutable cloud snapshot. It does not delete older snapshots.

Restore downloads the newest snapshot for the sync key. Before overwriting local files, DevLauncher creates timestamped local backups of the current `keyboard.yaml` and `quickmemory_data.json`. If writing either restored file fails, the command returns an error and leaves the backup files in place.

The first version does not merge local and remote changes. Restore is an explicit replace operation, protected by a confirmation prompt in the UI.

## Identity And Secrets

The user identity is a sync key.

The server never stores the raw sync key. It stores only a slow hash of the key. The client sends the raw key over HTTPS for API requests. On the desktop, the sync key is stored in the OS credential store after the user chooses to save it.

Recommended sync key format:

- At least 32 random bytes.
- Encoded as a user-copyable string, for example `dlsk_...`.
- Generated by the server or a setup command, then shown once to the user.

## Server Design

Run a separate sync API service from the static website container.

Suggested deployment units:

- `devlauncher-sync-api`
- PostgreSQL database
- HTTPS reverse proxy when the domain is ready

During early testing, the API can be bound to a non-public port or protected by firewall rules. Production use should require HTTPS.

PostgreSQL tables:

```sql
create table sync_keys (
  id uuid primary key,
  key_hash text not null unique,
  label text,
  created_at timestamptz not null default now(),
  last_used_at timestamptz
);

create table sync_snapshots (
  id uuid primary key,
  sync_key_id uuid not null references sync_keys(id) on delete cascade,
  device_name text,
  app_version text,
  schema_version integer not null,
  content_hash text not null,
  keyboard_config jsonb not null,
  quickmemory_data jsonb not null,
  created_at timestamptz not null default now()
);

create index sync_snapshots_key_created_idx
  on sync_snapshots(sync_key_id, created_at desc);
```

API endpoints:

- `POST /api/sync/snapshots`: authenticate with sync key and create a snapshot.
- `GET /api/sync/snapshots/latest`: authenticate with sync key and return latest snapshot metadata plus payload.
- `GET /api/sync/status`: authenticate with sync key and return latest snapshot metadata.

The API should validate payload shape, enforce a size limit, rate-limit failed key attempts, and avoid logging raw sync keys or snapshot payloads.

## Desktop Architecture

Add a Rust sync module that owns filesystem and credential access:

- Read `keyboard.yaml` through the existing config path.
- Read `quickmemory_data.json` through the existing QuickMemory data path.
- Save and load the sync key through OS credentials.
- Call the sync API with an explicit base URL.
- Create local backups before restore.

Add Tauri commands:

- `sync_get_status()`
- `sync_save_key(key, base_url)`
- `sync_upload_snapshot()`
- `sync_restore_latest_snapshot()`

The frontend should call these commands from `SettingsPanel.tsx`, keeping the UI thin. It should display progress, success, and failure states without exposing secret values.

## Data Shape

Snapshot payload:

```json
{
  "schemaVersion": 1,
  "deviceName": "MacBook Pro",
  "appVersion": "0.1.0",
  "keyboardConfig": {},
  "quickmemoryData": {}
}
```

The server stores `keyboardConfig` and `quickmemoryData` as JSONB, even though `keyboard.yaml` is YAML on disk. The client can convert loaded Rust structs to JSON for upload and write restored structs back through the existing serializers.

## Error Handling

Expected user-facing errors:

- Missing sync key.
- Invalid sync key.
- Server unavailable.
- Snapshot not found.
- Payload schema mismatch.
- Local backup failed.
- Local restore write failed.

Restore must fail closed. If the remote payload cannot be validated, DevLauncher does not touch local files.

## Security Notes

The first version is not end-to-end encrypted. That is acceptable only if the server is private, HTTPS is enforced, and no real passwords or private key contents are uploaded.

A later version can add client-side encryption where the sync key derives an encryption key and the server stores only ciphertext. That should be a separate design because it changes recovery and rotation behavior.

## Testing

Rust unit tests:

- Snapshot assembly excludes credential values.
- Restore validates payload before writing.
- Restore writes local backup files before overwriting.
- Missing files upload as default empty config/data where appropriate.

Frontend tests:

- Cloud Sync section shows disconnected and connected states.
- Upload and restore buttons handle loading and error states.
- Restore requires explicit confirmation.

Integration tests:

- Upload a snapshot to a local test API.
- Restore it into a temporary app data directory.
- Verify `keyboard.yaml` and `quickmemory_data.json` round-trip.

Manual checks:

- Existing configs with `pet.menu.customActions` still load and save correctly.
- URL and SSH actions with `hasPassword` restore without actual passwords.
- New computer restore surfaces a clear "passwords need re-entry" state.

## Open Follow-Ups

- Add historical snapshot selection after latest-only restore works.
- Add optional client-side encryption.
- Add plugin-specific sync adapters for API Lab, JSON helper history, and screenshots if they become durable user data.
- Add conflict comparison UI if realtime sync is ever introduced.
