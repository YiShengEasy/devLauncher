# Windows 主窗口控制统一方案

## 背景

Windows 版本当前同时通过 Tauri 窗口 API、异步 Win32 API 和键盘钩子线程控制主窗口。双击 Ctrl、红色关闭按钮、最小化按钮、托盘菜单和置顶按钮读取的窗口状态不完全一致，且双击 Ctrl 每次都会创建新的线程，因此快速连续操作时会出现显示、隐藏、置顶互相覆盖的问题。

本方案只统一 DevLauncher 主键盘窗口的控制路径。macOS 保留现有 Tauri 窗口行为，不引入 Win32 代码。

## 已确认的产品行为

- Windows 默认通过双击 Ctrl 切换主键盘窗口。
- 主窗口隐藏时，双击 Ctrl 显示并恢复窗口。
- 主窗口最小化时，双击 Ctrl 恢复窗口。
- 主窗口已经显示时，双击 Ctrl 隐藏窗口。
- 显示或恢复窗口后，窗口应进入当前屏幕前台。
- 右上角红色关闭按钮隐藏到托盘，进程继续运行，不退出应用。
- 黄色按钮只最小化窗口。
- 图钉按钮立即切换置顶状态并持久化。
- 图钉图标改为截图工具栏现有的简洁“定住”图标；共享该按钮的 Windows 和 macOS 界面保持一致。

## 范围边界

### 修改

- 主窗口的显示、隐藏、最小化、恢复和置顶控制入口。
- Windows 双击 Ctrl 触发窗口切换的投递方式。
- 主窗口红色关闭按钮和黄色最小化按钮的调用路径。
- Windows 托盘和单实例唤起主窗口时的调用路径。
- Windows 置顶状态应用方式。
- 共享图钉图标。

### 保留

- macOS 的全局快捷键、窗口定位、窗口层级和 Tauri 控制逻辑。
- 现有置顶状态存储格式和前端状态事件。
- 当前工作区中与截图复制、标题拖动、阴影和快捷键提示有关的未提交改动。
- 其他插件窗口现有的关闭和最小化行为，除非它们复用共享图钉按钮。

### 延后

- 重构全部插件窗口的窗口控制。
- 修改快捷键配置界面或增加新快捷键。
- 自动化窗口交互测试。

## 方案选择

### 方案 A：统一动作入口，平台分别实现（采用）

前端、键盘钩子、托盘和单实例事件只提交语义化动作；Windows 使用串行化的 Win32 控制器，macOS 继续调用现有 Tauri API。该方案能消除状态源不一致和并发竞争，同时将平台差异限制在后端模块内。

### 方案 B：全部改为 Tauri 窗口 API

代码更少，但当前 Windows 上已经出现前端 Tauri 状态与原生窗口状态不同步的问题，无法可靠解决最小化恢复、前台激活和快速切换。

### 方案 C：各入口继续直接调用 Win32

局部修改较快，但键盘钩子、按钮和托盘仍会各自查询、修改状态，竞态条件继续存在，后续维护成本也更高。

## 架构

新增一个职责单一的后端窗口控制模块，对调用方暴露统一动作：

- `ToggleMain`：根据真实窗口状态显示或隐藏主窗口。
- `ShowMain`：显示或恢复主窗口并尝试置前。
- `HideMain`：隐藏到托盘，保持进程运行。
- `MinimizeMain`：最小化主窗口。
- `SetPinned`：应用并持久化置顶状态。

所有动作进入单一串行队列。队列只允许一个执行者按顺序处理窗口动作，键盘钩子不再为每次双击创建独立窗口控制线程。

### Windows 实现

Windows 分支以 HWND 的原生状态为唯一运行时事实来源：

- 使用 `IsWindowVisible` 判断是否隐藏。
- 使用 `IsIconic` 判断是否最小化。
- 使用同步 `ShowWindow` 完成隐藏、最小化和恢复，避免异步动作乱序。
- 恢复后通过 `BringWindowToTop` 和 `SetForegroundWindow` 请求前台显示。
- 置顶通过 `SetWindowPos(HWND_TOPMOST/HWND_NOTOPMOST, ...)` 应用，不再混用 Tauri 的置顶状态。

Win32 代码只存在于 `#[cfg(target_os = "windows")]` 分支中。

### macOS 和其他平台实现

macOS 分支继续使用现有的 Tauri `show`、`hide`、`minimize`、`set_focus` 和 `set_always_on_top` 行为。统一动作入口只提供调用接口，不改变 macOS 的底层实现、快捷键和窗口生命周期。

其他非 Windows、非 macOS 平台使用相同的 Tauri 回退实现。

## 调用路径

```text
双击 Ctrl ─┐
红色按钮 ─┤
黄色按钮 ─┼─> 统一窗口动作入口 -> 串行队列 -> Windows Win32 实现
托盘菜单 ─┤                              └-> macOS/其他平台 Tauri 实现
单实例唤起 ┘
```

前端按钮不再直接调用 `getCurrentWindow().hide()` 或 `minimize()` 控制主窗口，而是调用后端语义化命令。这样按钮、双击 Ctrl 和托盘不会再维护不同的窗口状态判断。

## 状态转换

| 当前状态 | 动作 | 结果 |
| --- | --- | --- |
| 隐藏 | `ToggleMain` / `ShowMain` | 恢复、显示并请求前台 |
| 最小化 | `ToggleMain` / `ShowMain` | 恢复并请求前台 |
| 正常显示 | `ToggleMain` | 隐藏到托盘 |
| 任意 | `HideMain` | 隐藏到托盘，进程继续 |
| 正常显示 | `MinimizeMain` | 最小化 |
| 任意 | `SetPinned(true)` | 立即置顶并持久化 |
| 任意 | `SetPinned(false)` | 取消置顶并持久化 |

显示或恢复窗口时，控制器会读取已持久化的置顶状态并重新应用，保证恢复后层级正确。

## 图钉图标

将截图工具栏当前使用的简洁图钉 SVG 提取到现有共享图标模块，`WindowPinButton` 和截图工具栏共同复用它。只统一图标几何和线条风格，不改变按钮尺寸、点击区域、悬停反馈或置顶状态颜色。

## 错误处理

- 动作入口找不到主窗口时返回明确错误，不修改持久化状态。
- Win32 调用失败时记录动作类型和系统错误，队列继续处理后续动作，避免控制器永久失效。
- 前台激活受 Windows 系统策略限制时，窗口仍保持已恢复和可见；激活失败不回滚显示状态。
- 置顶持久化只在窗口层级应用成功后更新，避免界面显示已置顶但系统窗口未置顶。

## 实施后的检查边界

按用户要求，交互效果由用户手动测试。实现阶段不自动操作窗口，不执行双击 Ctrl、关闭、最小化或置顶的交互测试；只进行代码审查、UTF-8/LF 检查，以及必要的编译检查，确保 Windows 修改没有进入 macOS 专用分支。

## 验收标准

- Windows 上反复双击 Ctrl 不会因为并发窗口动作而随机失效。
- 隐藏和正常显示状态可以稳定互相切换。
- 最小化状态可以通过双击 Ctrl 恢复。
- 显示或恢复后窗口出现在前台；置顶开启时保持最前。
- 红色按钮始终隐藏到托盘且进程继续运行。
- 黄色按钮可以最小化。
- 图钉按钮立即生效，重启后状态仍保留。
- macOS 不包含 Win32 调用，原有窗口行为保持不变。
