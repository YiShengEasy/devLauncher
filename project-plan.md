## 一、技术方案扩展性评估

### 总体结论

> **扩展性：强 ✅ | 可行性：高 ✅**

V1 技术方案在核心抽象层面设计合理，以 Action 为统一模型、Tauri 2 作为基座，天然支持 V2/V3 的演进。以下是逐项分析：

---

### 各项技术评分

| 技术选型 | 扩展性 | 风险 | 说明 |
|---|---|---|---|
| **Tauri 2** | ⭐⭐⭐⭐⭐ | 低 | 原生插件系统（`tauri-plugin-*`），Rust 后端可低成本扩展新能力 |
| **Action 抽象模型** | ⭐⭐⭐⭐⭐ | 低 | 统一接口设计，新动作类型只需实现接口，不影响已有功能 |
| **React 19 + TypeScript** | ⭐⭐⭐⭐⭐ | 低 | 组件化 + 强类型，UI 扩展成本低，插件 UI 可注册为组件 |
| **Zustand** | ⭐⭐⭐⭐ | 低 | Slice 模式支持模块化状态扩展，不同功能模块互不干扰 |
| **shadcn/ui** | ⭐⭐⭐⭐ | 低 | 无锁定设计，组件完全可定制，适合长期演进 |
| **dnd-kit** | ⭐⭐⭐⭐ | 低 | 模块化拖拽，V2 流程编排也可复用 |
| **YAML 配置** | ⭐⭐⭐ | 中 | V1/V2 够用，V3 自动化流程复杂度会使 YAML 难以维护，需规划迁移至 SQLite |
| **内置终端（SSH）** | ⭐⭐⭐⭐ | 中 | 内置终端完全掌控体验，跨平台一致，但需实现 PTY 伪终端和 SSH 协议，开发成本较高 |

---

### 扩展性优势

**1. Action 接口天然支持插件化**

```
V1: App / SSH / Script / Folder / File / Url
V2: ActionChain（串联多个 Action）
V3: PluginAction（第三方注册）
```

只需定义统一的 `Action` 接口，新类型扩展零破坏性。

**2. Tauri 2 插件架构对齐 V3 目标**

Tauri 2 的 `tauri::plugin::Builder` 模式可直接用于 V3 插件系统，现在写的 Rust command 未来可直接包装为 plugin。

**3. Page/Key/Action 三层模型是稳定的核心**

三层解耦意味着：UI 层（Key 渲染）、配置层（Page 管理）、执行层（Action 触发）可以独立演进。

---

### 风险与建议

**风险 1：YAML 配置的复杂度天花板**

- 问题：V3 的动作链（A→B→C）、条件分支在 YAML 中表达笨拙
- 建议：V1 用 YAML，V2 引入 `tauri-plugin-sql`（SQLite），Action 配置迁入数据库，YAML 仅保留用户设置

**风险 2：内置终端开发成本**

- 方案：内置终端替代 `wt.exe`，体验统一，无外部依赖
- 技术路线：Tauri 侧使用 `portable-pty`（Rust crate）驱动 PTY，SSH 协议使用 `russh` 或 `ssh2` crate，前端用 `xterm.js` 渲染
- 风险点：PTY + xterm.js 集成工作量大，建议 V1 先用 `wt.exe` 兜底快速验证，V1.x 或 V2 迭代为内置终端

**风险 3：V3 插件 API 边界未定义**

- 问题：如果现在不设计插件合约，V3 重构成本会很高
- 建议：V2 阶段同步设计 `PluginManifest` 规范（即便 V2 不实现插件）

**风险 4：架构图缺少插件扩展点**

- 建议：V2 开发前补充扩展后的架构图，明确 Plugin Registry 的位置
